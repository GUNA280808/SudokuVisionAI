# AI-Powered Sudoku Vision Solver 🧩

Upload a photo of a Sudoku puzzle, and this app will detect the grid, recognize
the digits with a CNN, solve it with a backtracking algorithm, score its
difficulty, and let you download a full PDF report — all through a
Bootstrap 5 dashboard with dark mode and webcam capture.

## Tech Stack

| Layer | Tech |
|---|---|
| Frontend | HTML5, CSS3, JavaScript, Bootstrap 5 |
| Backend | Python, Flask |
| Computer Vision | OpenCV |
| Digit Recognition | TensorFlow/Keras CNN (trained on MNIST) |
| Solving Algorithm | Backtracking |
| PDF Reports | ReportLab |
| Database | SQLite |
| Deployment | GitHub → Render |

## 1. Local Setup

```bash
git clone <your-repo-url>
cd SudokuVisionAI

python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

pip install -r requirements.txt
```

## 2. Train the CNN (already done for you, but here's how to redo it)

A trained model is already included at `models/cnn_model.h5`, so you can skip
straight to step 3. If you ever want to retrain it (e.g. after tweaking the
CNN architecture or the segmentation pipeline), regenerate the dataset and
retrain:

```bash
python generate_printed_digits.py   # builds a synthetic printed-digit dataset
python train_model.py               # trains and saves models/cnn_model.h5
```

**Why printed digits instead of MNIST:** real Sudoku puzzles (newspapers,
apps, books, screenshots) use printed/typed digits, not handwriting. A CNN
trained on MNIST (handwritten digits) generalizes poorly to printed fonts
and misreads digits constantly, which shows up as "duplicate digit"
validation errors on nearly every puzzle. `generate_printed_digits.py`
renders digits 1-9 in ~18 real system fonts with random rotation, blur,
and noise, and — critically — normalizes each glyph's bounding box to a
canonical size before training (the same normalization the live pipeline
applies at inference time in `vision/segmentation.py::normalize_digit_glyph`).
Matching that scale/centering between training and inference was the fix
for the "every puzzle fails" bug: without it, the model learns digits at
one scale but sees them at another in production. No internet access is
required — everything is generated locally from system fonts.

> Digit "0" / blank cells are handled separately by the empty-cell-detection
> step in `vision/segmentation.py`, so the CNN only ever classifies 1-9.

## 3. Run Locally

```bash
python app.py
```

Visit `http://127.0.0.1:5000`.

## 4. Deploy to Render

1. Push this repo to GitHub (make sure `models/cnn_model.h5` is committed,
   or train it as part of your Render build — see note below).
2. On [Render](https://render.com), create a **New Web Service** from your
   GitHub repo.
3. Settings:
   - **Build Command:** `pip install -r requirements.txt`
   - **Start Command:** `gunicorn app:app --bind 0.0.0.0:$PORT --timeout 120`
   - **Environment:** Python 3.11
4. Add an environment variable `SECRET_KEY` with a random string.
5. Deploy — Render will give you a public URL.

> **Model file note:** `cnn_model.h5` is a few MB and is fine to commit to
> Git directly. If your Git host rejects binary files that size, use Git LFS,
> or run `generate_printed_digits.py` + `train_model.py` as part of your
> Render build step instead (no internet access needed — the training data
> is generated locally from system fonts already present in the build image;
> if a font search comes up empty, install `fonts-dejavu-core`,
> `fonts-liberation`, and `fonts-freefont-ttf` first).

## How the Pipeline Works

1. **Upload / Capture** — image comes in via file upload or browser webcam.
2. **Validation** — extension, file size, and resolution are checked.
3. **Preprocessing** (OpenCV) — resize → grayscale → Gaussian blur →
   adaptive threshold → edge detection.
4. **Grid Detection** — largest contour is assumed to be the Sudoku border;
   it's reduced to 4 corners.
5. **Perspective Correction** — the grid is warped into a flat, square image.
6. **Segmentation** — the square is split into an 81-cell grid; each cell
   is checked for emptiness.
7. **Digit Recognition** — non-empty cells are classified 1-9 by the CNN.
8. **Validation** — the recognized grid is checked for rule violations
   (duplicate digits in a row/column/box) before solving.
9. **Solving** — a backtracking algorithm fills in the grid, counting
   recursive steps and timing itself.
10. **Difficulty Scoring** — a weighted score (clue count + backtracking
    steps + solve time) classifies the puzzle as Easy / Medium / Hard / Expert.
11. **Overlay & Output** — the solved digits are drawn back onto the
    original grid image, saved, stored in SQLite, and compiled into a
    downloadable PDF report.

## Tuning Notes

- `vision/segmentation.py: is_cell_empty()` — the `white_pixel_threshold`
  and center-crop ratio may need slight recalibration depending on your
  camera's lighting and the pen/print contrast in your test photos.
- `vision/contour.py` — if grid detection fails on very low-contrast photos,
  try adjusting the adaptive threshold block size/C constant in
  `preprocess.py`.
- `solver/difficulty.py` — the weights (`W_CLUES`, `W_STEPS`, `W_TIME`) and
  normalization anchors can be tweaked to match how "hard" you want puzzles
  to feel in the Easy/Medium/Hard/Expert buckets.
